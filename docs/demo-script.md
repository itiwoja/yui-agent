## 先回りすべき審査員の一言

**「タスク登録だけなら普通の Gemini でよくね？」**

その通りなので、デモは「登録が便利」を見せる構成にしない。見せるのは登録の外側:

1. **頼んでいないのに拾う** — 「タスクにして」と言っていない雑談から拾う（登録という行為の消滅）
2. **記憶が育つ** — 数日前の言及が再言及で優先度昇格し、放置タスクが🔴に育っている
3. **勝手に働く** — 誰も話しかけていない30分の間に、調査ノートと質問が増えている
4. **完了まで見届ける** — 「終わったよ」の一言で Firestore と Google Tasks の両方が閉じる

この4つのうち 2 と 3 は**デモ当日に仕込みが必要**（後述のチェックリスト参照）。
当日の新規登録だけでは「普通のGemini」との差が見えない。

## デモの流れ（5〜7分 / 1日のストーリー仕立て）

### 0. 準備（画面に出す前に）

- ブラウザで本番URLを開き「そばに置く」を ON（マイク許可済みの状態にしておく）
- 別タブ: ダッシュボード / GitHub Actions / スマホに Google Tasks アプリ
- **数日前から仕込んでおく**: 同じタスク（例: 請求書）に2〜3回言及して優先度を育て、
  放置タスクの経過時間バッジ（「3日前」= 注意色）が出ている状態を作っておく

### 1. つかみ — 育った記憶を先に見せる（約1.5分）

- **いきなりダッシュボードから開く**（新規登録からではなく）:
  - 数日前の言及が再言及で**優先度昇格**している（🔴 + 「言及3回」バッジ）
  - **放置タスクの経過時間**（「5日前」が警告色）が古い順に並んでいる
  - ゆいが自動で添付した**裏どり調査ノート**と「**ゆいから質問があります**」
- 語り: 「これ、全部『入力』していません。数日分の独り言と雑談から、ゆいが勝手に
  拾って、覚えて、育てて、調べて、質問してきた結果です」
- → 「普通のGeminiでよくね？」を最初の90秒で潰す

### 2. 朝 — 雑談から拾う（約1分）

- 画面に向かわず、作業しながら雑談するように:
  - 「あー、**Webデザインの課題**、金曜までだった気がするんだよなー」
- → ゆいが短く相槌を返し（音声）、会話の流れでタスク化される
  （ウェイクワード不要。「タスクにして」とも言っていない点を口頭で強調）
- ダッシュボードを更新して「頼んでいないのに拾われている」を見せる

### 3. 昼 — 相談（マルチツールの見せ場・約1.5分）

- 「今日なにからやればいい？」
- → ゆいが **Google Calendar の今日の予定＋Firestore の未完了タスク優先度** を合わせて提案
  （例: 「15時から授業だよね。その前に課題の下書きやっちゃお？」）
- 続けて: 「そういえば副業の**請求書**も出さなきゃなんだよね」→ 会話からタスク追加

### 4. 裏側 — エージェントが勝手に働く（約1.5分）

- `POST /autonomous-review` を手動トリガー（本番は Cloud Scheduler が30分毎）
- ダッシュボードで見せる:
  - 放置タスクの**優先度エスカレーション**がその場で起きる
  - Google Search grounding による**調査ノート**の新規添付
  - 「ゆいから質問があります」に**ダッシュボードから回答**→ タスクが前進する
    （ask → answer → 再開のループが閉じることを見せる）
- スマホの Google Tasks にも同期されていることを見せる

### 5. 夕方 — 「終わったよ」でループが閉じる（約1分）

- 「**Webデザインの課題終わったよ**」
- → ゆいがねぎらう＋Firestore と Google Tasks の**両方が完了**になる
- ダッシュボードの完了一覧で確認

### 6. 締め — 「まわしている」を見せる（約1.5分）

- GitHub Actions のタブ: **push → check（依存解決+構文検証）→ Cloud Run デプロイ** のパイプライン
- Cloud Run のリビジョン履歴（自動デプロイの証跡）
- アーキテクチャ一言まとめ:
  - Cloud Run + Vertex AI (Gemini 3.5 Flash) + Speech-to-Text + TTS (Chirp3-HD) + Firestore + Cloud Scheduler + Secret Manager + Cloud Trace + WIF キーレスCI/CD

## 想定問答

**Q. タスク登録だけなら普通の Gemini（チャットアプリ）でよくない？**
A. 一発の「発話→登録」だけならその通りです。ゆいの価値は登録の外側にあります。
(1) 「タスクにして」と頼む意図すら不要 — 本人がタスクだと認識していない雑談から拾う、
(2) セッションを跨いだ記憶 — 表記揺れを超えた再言及検出で優先度が育ち、放置が可視化される、
(3) 自律ループ — 誰も話しかけていない間に調査・下書き・質問を勝手に進める、
(4) 実行系接続 — Google Tasks / Calendar に直結し「終わったよ」で完了まで閉じる。
チャットは「聞けば答える」、ゆいは「言わなくても動く」です。

**Q. なぜ ADK（Agent Development Kit）を使わなかった？**
A. エージェントの中核が「Sense→Diagnose→Act/Ask」という確定的な制御フローで、Gemini の構造化出力＋明示的な分岐で書く方がデバッグ容易性と挙動の説明可能性が高いと判断しました。ツールが増えるフェーズ（Calendar 書き込み・Gmail 等）で ADK のツール管理・セッション管理に載せ替えるのがロードマップです。

**Q. プライバシーは？ 常時マイクは怖くない？**
A. 音声はブラウザ内で VAD 判定し、発話区間のみサーバーへ送信。閾値・無音判定はクライアントで調整可能です。データは本人の Firestore / Google Tasks にのみ保存され、OAuth トークンは Secret Manager 管理、Cloud Run はランタイムSAの最小権限で動きます。

**Q. 全部の発話に反応するなら、雑談を誤ってタスク化しない？**
A. 抽出は確信度つきの構造化出力で、閾値（既定0.6）未満は登録しません。プロンプトでも「発言に実際に含まれる内容だけを根拠に、曖昧なら聞き返す」と縛っており、音声認識は最高精度の Chirp 2 を使っています。無音判定などの環境依存パラメータは URL クエリで会場に合わせて即調整できます。

**Q. 常時マイクで全発話を Gemini に送ると、コストが青天井では？**
A. ブラウザ側 VAD で発話区間のみ送信し、サーバ側はレート制限・入力サイズ上限・タスク件数上限（自律ループも件数制限つき）で防御しています。エンドポイントはトークン認証で保護され、本番はトークン未設定だと起動自体が失敗する設計です。

**Q. 既存のタスクアプリ・Circleback 等との違いは？**
A. 既存ツールは「入力されたもの」を管理します。ゆいは「入力されなかったもの」を発見し、優先度を育て、質問し、完了まで見届けます。管理ツールではなく秘書です。

## リハーサル・チェックリスト（本番同環境で最低3回）

本番と同じ PC・マイク・ネットワーク・会場の騒音条件で通しリハーサルする。

- [ ] **数日前から**: 「つかみ」用のデータ仕込み。同じタスクに日を分けて2〜3回言及し、
      優先度昇格・言及回数・放置バッジ・調査ノート・保留質問が揃った状態を作る
      （当日に作れないデモ素材はこれだけ。最重要）
- [ ] リハ1回目: 素の状態で台本を通す。誤検出・無音判定の切れ方・応答の間をメモ
- [ ] パラメータ調整: `?silence=1800&threshold=12` のように URL クエリで調整して再試行
  - `silence` (既定2200ms): 早く切れる→上げる / テンポが悪い→下げる
  - `threshold` (既定10): 騒がしい会場→上げる / 拾いが悪い→下げる
  - `minSpeech` (既定500ms) / `maxRecord` (既定20000ms)
- [ ] リハ2回目: 調整後の値で通す。TTS再生中にオーブタップで停止→復帰も1回試す
- [ ] リハ3回目: 決定値で最初から最後までノーミスで通す
- [ ] 決定値をコードのデフォルト値に反映して push（→ CI/CD で自動デプロイ）
- [ ] 当日朝: `/autonomous-review` の手動トリガーと Google Tasks 同期を最終確認
